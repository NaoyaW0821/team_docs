# 思想と原則

ここには、技術と運営を貫くチームの共通原則を記述する。
細かい仕様や数値はなるべく省くことを努力する。


## 基本思想
- このチームは何を最優先するか
  - 優勝(≒打倒・松本工業)
  - この目標は、実はチームリーダが独断で決めてもいい。

- 勝ち方の考え方
  - 車両にかかる抵抗をリストアップし、お金をかければ修正できる部分と技術力がないと修正できない部分に分ける。
  - [Endless Sphere DIY EV Forum](https://endless-sphere.com/sphere/) とか、[Redditの技術系板](https://www.reddit.com/r/AskEngineers/) とかで抵抗を減らすテクニックなどを調べる。ChatGPTやGeminiのDeepResearchを併用するとベスト。


## 技術に関する原則
- 軽量化とは何か
  - 転がり抵抗を減らすためだけの行為じゃない
  - 車両の**複雑性**、**加工容易性**、**再現性**にも効いてくるので、積極的に推し進めるべき。

なので、こういう軽量化はしないようにする。
  - 信頼性を失わせる軽量化
  - ピーク性能のみを重視し、安定性や再現性を捨てた軽量化

**50%の確率で壊れる高性能マシンは望ましくないよ！**

- 軽量化の本質
  - 使われない剛性
  - 使われない強度
  - 使われないマージン
  - 使われない機構

  ちなみにこれだけです。 でも、これ捨てるのちょっと難しいよね。がんばって。

- 効率・ピーク・再現性の優先順位
**ピーク性能 < 効率 < 再現性**

  - 再現性
    - 同じ条件で同じ結果が出ること
    - だれが運転しても大きく性能が変わらないこと
    - 構造、制御、センサー類がある程度どんな環境でも安定して動作すること

  - 効率
    - 巡航速度までの加速に必要なエネルギー量 [W]
    - 入力（工数・コスト）に対する成果

  - ピーク奥率
    - 最高速
    - 最大加速度
    - 限界挙動
   
- 軽量化するときのチェック要素
  - 何を削るのか
  - なぜ削って問題ないのか
  - 削った結果、どの性能が向上するのか
  - 再現性への影響はあるか
  - 将来の設計変更で破綻しないか
 
- 完璧 VS 納期
  - 基本的に、 **未完成でも動くもの ＞ 完璧だが存在しないもの**
  - 完成していなくてもいったん提出・報告する。
  - というか3割、6割できた前提で上に一度確認をとる。初版は40点を超えていればOK
 
  以下の条件を満たす場合のみ、完成度を優先してよい。
  - 締切に影響しない
  - 他の作業を止めない
  - 再現性または安全性に直結する

- 判断、決断
  - 判断は先送りしない (0.4秒で考えたことと2時間で考えたことは大差ない場合が多い)
  - 間違った決定は修正できるが、遅延は修正できない
  - 決断前に一度判断を人に見てもらうと楽

## 運営に関する原則
- 運営の前提
  以下のことを前提にしないと破綻しがち。
  - 人は常に足りない
  - 時間は常に足りない
  - 情報共有は自然には起きない
  - 勝手に人は動かない
  - 他人のモチベーションは0である
    
人員増加を前提にした計画は破綻したときつらいので、今いる人間だけで成立するかを常に判断基準とする。

- タスクのリスク分散
  - 専任担当が存在しないことを前提にする
  - 属人化を排除するのではなく、ブラックボックス化を排除
  - 担当が抜けても回るくらいが理想
  - 判断基準、設計意図、失敗した案とその理由は文書化する。

- 運営判断の評価基準
  - 運営上の判断は、以下の観点で評価する。
  - チーム全体の作業は前に進んだか
  - 属人化せず引き継げるか
  - 締切に対して安全側か
  - 後から「なぜそうしたか」説明できるか
